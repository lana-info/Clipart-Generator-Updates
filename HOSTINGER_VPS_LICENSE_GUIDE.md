# Инструкция: сервер лицензий на Hostinger VPS для новичка

Эта инструкция написана **максимально простым языком**, чтобы вы могли идти по шагам, даже если раньше **никогда не настраивали сервер**.

Она подходит, если вы будете размещать сервер лицензий на **Hostinger VPS / Cloud VPS**.

Мы будем делать всё постепенно:
- сначала запустим сервер лицензий просто по IP;
- потом подключим домен;
- потом включим HTTPS;
- потом проверим, что приложение умеет проверять ключи.

Ниже описан самый понятный и нормальный путь:
- сервер лицензий на `FastAPI`
- запуск через `systemd`
- проксирование через `Nginx`
- HTTPS через `Let's Encrypt`
- хранение базы лицензий в файле `licenses.db`

---

## 0. Очень коротко: что мы вообще делаем

У вас есть приложение, которому нужно проверять лицензионные ключи.

Для этого нужен **отдельный маленький сервер**, который будет:
- хранить ключи;
- отвечать: ключ правильный или нет;
- запоминать, на каком устройстве ключ уже активирован;
- создавать новые ключи для клиентов.

Этот сервер будет стоять на вашем VPS.

Если говорить совсем просто:

- **VPS** = ваш удалённый компьютер в интернете;
- **SSH** = способ зайти в этот компьютер через терминал;
- **домен** = красивый адрес вместо IP;
- **Nginx** = программа, которая принимает запросы с сайта и передаёт их вашему серверу лицензий;
- **HTTPS** = защищённое соединение с замочком;
- **systemd** = способ сделать так, чтобы сервер запускался автоматически сам.

Если какие-то слова пока непонятны — это нормально. Просто идите по шагам.

## 1. Что будет в итоге

После настройки у вас будет работать адрес примерно такого вида:

```text
https://license.вашдомен.com
```

Сервер будет уметь:
- активировать ключи
- проверять ключи
- деактивировать ключи
- создавать ключи через админ-эндпойнты
- показывать список ключей

Админ-эндпойнты будут защищены токеном:
- `POST /admin/generate`
- `GET /admin/list`

Итоговая схема будет такая:

1. приложение отправляет запрос на ваш сервер;
2. сервер смотрит в базу `licenses.db`;
3. сервер отвечает, можно ли активировать/проверить ключ;
4. приложение либо разрешает работу, либо показывает ошибку.

## 11. Настройка переменных окружения

Сервер лицензий использует такие переменные:

- `LICENSE_DB_PATH`
- `LICENSE_ADMIN_TOKEN`

### Что они значат

#### `LICENSE_DB_PATH`
Путь к базе лицензий.

Рекомендую:

```text
/opt/clipart-license-server/data/licenses.db
```

#### `LICENSE_ADMIN_TOKEN`
Секретный токен для защиты админ-эндпойнтов.

Пример:

```text
CG_ADMIN_9fK2xP7mQ4vL8sR1zN6tW3yH5uB0aD
```

### Объяснение простыми словами

Этот токен — как секретный пароль для админки.

Без него нельзя:
- создавать ключи;
- смотреть список ключей.

То есть даже если кто-то узнает адрес сервера, он не сможет пользоваться админ-методами без токена.

## 13. Создание systemd-сервиса

Теперь сделаем так, чтобы сервер запускался автоматически.

Создайте файл:

```bash
nano /etc/systemd/system/clipart-license.service
```

Вставьте туда:

```ini
[Unit]
Description=Clipart License Server
After=network.target

[Service]
User=root
WorkingDirectory=/opt/clipart-license-server
Environment="LICENSE_DB_PATH=/opt/clipart-license-server/data/licenses.db"
Environment="LICENSE_ADMIN_TOKEN=ВАШ_СЕКРЕТНЫЙ_ТОКЕН"
ExecStart=/opt/clipart-license-server/.venv/bin/uvicorn license_server:app --host 127.0.0.1 --port 8000
Restart=always
RestartSec=3

[Install]
WantedBy=multi-user.target
```

Сохранить:
- `Ctrl + O`
- `Enter`
- `Ctrl + X`

### Важно
Вместо:

```text
ВАШ_СЕКРЕТНЫЙ_ТОКЕН
```

вставьте ваш реальный токен.

### Почему это важно

Если оставить текст `ВАШ_СЕКРЕТНЫЙ_ТОКЕН`, сервер стартует, но вы потом не сможете нормально пользоваться админкой, потому что токен будет неправильный.

Лучше сразу придумать длинный токен из букв и цифр.

Например:

```text
CG_ADMIN_4mP8xR2vT7kL9sW1nQ6yH3zD5uB0cF
```

## 17. Настройка Nginx

Создайте конфиг:

```bash
nano /etc/nginx/sites-available/clipart-license
```

Вставьте:

```nginx
server {
    listen 80;
    server_name license.вашдомен.com;

    location / {
        proxy_pass http://127.0.0.1:8000;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
    }
}
```

Сохраните файл.

Теперь включите сайт:

```bash
ln -s /etc/nginx/sites-available/clipart-license /etc/nginx/sites-enabled/clipart-license
```

Проверьте конфиг:

```bash
nginx -t
```

Если всё хорошо, перезапустите Nginx:

```bash
systemctl reload nginx
```

### Что делает Nginx

Он принимает запросы с адреса:

```text
https://license.вашдомен.com
```

и передаёт их вашему Python-серверу, который работает внутри VPS на:

```text
127.0.0.1:8000
```

То есть снаружи виден красивый адрес, а внутри работает ваш `uvicorn`.

## 18. HTTPS через Let's Encrypt

Когда домен уже смотрит на VPS, выполните:

```bash
certbot --nginx -d license.вашдомен.com
```

Дальше certbot задаст несколько вопросов.

Обычно для новичка достаточно:
- указать email
- согласиться с условиями
- выбрать redirect на HTTPS

После этого сайт начнёт работать по HTTPS.

### Очень важно

Сертификат не выпустится, если:
- домен ещё не направлен на сервер;
- Nginx настроен с ошибкой;
- 80 порт закрыт firewall.

Поэтому сначала проверьте, что сайт открывается по `http`, а уже потом запускайте `certbot`.

## 20. Как создавать ключи через защищённую админку

Теперь админка защищена токеном.

### Создание ключей

Пример:

```bash
curl -X POST "https://license.вашдомен.com/admin/generate" \
  -H "Content-Type: application/json" \
  -H "X-Admin-Token: ВАШ_СЕКРЕТНЫЙ_ТОКЕН" \
  -d '{
    "count": 5,
    "prefix": "CG",
    "mode": "random",
    "expires_days": 365,
    "updates_days": 365,
    "max_devices": 1,
    "serial_start": 1
  }'
```

### Что делает эта команда

Она создаёт 5 новых лицензионных ключей:
- префикс `CG`
- срок действия 365 дней
- обновления 365 дней
- 1 устройство на ключ

### Что вы получите в ответ

Сервер вернёт JSON, где будут реальные ключи.

Их нужно сохранить в безопасное место.

### Посмотреть список ключей

Через query-параметр:

```text
https://license.вашдомен.com/admin/list?admin_token=ВАШ_СЕКРЕТНЫЙ_ТОКЕН
```

Или через заголовок — это безопаснее.

### Что я советую вам лично

Для просмотра списка можно иногда использовать query-параметр, это проще.

Но для создания ключей лучше использовать именно заголовок `X-Admin-Token`.

## 21. Что вписать в приложение

В `config.json` приложения нужно указать:

```json
{
  "license_server_url": "https://license.вашдомен.com",
  "license_backup_server_url": ""
}
```

Если файл уже содержит другие настройки — просто замените эти 2 поля.

### Важно

В `license_server_url` нужно писать адрес **без** `/health`, `/validate` и других хвостов.

Правильно:

```text
https://license.вашдомен.com
```

Неправильно:

```text
https://license.вашдомен.com/health
```

## 23. Как проверить всё целиком

Порядок проверки:

1. Открыть:

```text
https://license.вашдомен.com/health
```

2. Создать 1 тестовый ключ через `/admin/generate`
3. Вписать `license_server_url` в `config.json`
4. Запустить приложение
5. Ввести ключ
6. Нажать проверку/активацию лицензии

Если всё хорошо — значит связка работает.

### Какой порядок самый правильный

Вот прямо идеальный порядок для вас:

1. сервер стартует локально на VPS;
2. `curl http://127.0.0.1:8000/health` работает;
3. Nginx настроен;
4. `http://license.вашдомен.com/health` работает;
5. HTTPS включён;
6. `https://license.вашдомен.com/health` работает;
7. создаётся тестовый ключ;
8. приложение проверяет этот ключ.

Если идти именно в таком порядке — почти всегда легче понять, где ошибка.
